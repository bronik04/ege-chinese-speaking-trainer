# ADR 0003: Единая стратегия миграций базы данных

**Статус:** Accepted

## Контекст

Исторически SQLite обновлялся функциями из `sqlite_migrations.py`, а PostgreSQL — опубликованной цепочкой Alembic. Параллельное добавление схемы в оба механизма создаёт риск расхождения production и локальной базы.

## Решение

Alembic является каноническим владельцем всех новых изменений схемы для SQLite и PostgreSQL. Опубликованные ревизии до `20260711_03` и SQLite-миграции 1–7 образуют неизменяемый baseline.

При первом запуске SQLite compatibility path применяет baseline 1–7, проверяет полный список версий, штампует базу ревизией `20260711_03` и выполняет `upgrade head`. PostgreSQL выполняет `upgrade head` напрямую. Все последующие изменения добавляются одной новой Alembic revision, поддерживающей оба dialect.

Dialect-specific SQL допускается внутри одной revision через проверку `op.get_bind().dialect.name`. Для SQLite следует использовать Alembic batch operations, когда операция не поддерживается SQLite напрямую.

## Правила разработки

- Не изменять и не переупорядочивать SQLite baseline 1–7.
- Не редактировать уже опубликованные Alembic revisions.
- Не добавлять новую схему одновременно в SQLite baseline и Alembic.
- Для новой revision проверять чистую базу, обновление существующей базы и повторный `upgrade head` на SQLite и PostgreSQL.
- Проверять паритет таблиц, колонок и именованных индексов обеих СУБД в CI.
- Изменение PostgreSQL также должно пройти backup/restore smoke test.

## Последствия

SQLite сохраняет совместимость с существующими локальными базами без переписывания истории. Следующая миграция имеет одну каноническую реализацию, а CI обнаруживает расхождение схем до развёртывания.
