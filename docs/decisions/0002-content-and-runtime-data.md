# ADR 0002: Границы контента и runtime-данных

**Статус:** Accepted

## Контекст

Сейчас версионируемые JSON-материалы находятся в `data/`, публичные изображения — в `assets/`, а локальная БД, аудио и material assets — в исключённом из Git `var/`. Названия не полностью показывают, какие данные можно публиковать, коммитить и безопасно удалять.

## Решение

Целевая структура использует следующие категории:

- `content/` — версионируемые учебные материалы, выдаваемые только через разрешённые API;
- `public/` — версионируемые файлы, предназначенные для прямой выдачи браузеру;
- `var/` — некоммитируемые локальные runtime-данные и пользовательские файлы;
- `backups/` — некоммитируемые резервные копии;
- `tmp/` — некоммитируемые воспроизводимые промежуточные артефакты.

Физический перенос из `data/` и `assets/` выполняется отдельной задачей с сохранением публичных URL и тестами доступа.

## Последствия

- Назначение файлов становится очевиднее человеку, CI и AI-агенту.
- Можно независимо определять backup, retention и access policy для runtime-данных.
- Content validation можно запускать без доступа к пользовательским данным.
- До миграции документация указывает как текущее, так и целевое имя каталога.

## Ограничения

- `content/` не означает публичный файловый доступ: авторизация остаётся на API.
- `public/` не должен содержать секреты, персональные данные или пользовательские загрузки.
- `tmp/` не используется как постоянное хранилище или единственная копия результата.
- Backup и аудио остаются закрытыми независимо от используемого storage backend.
