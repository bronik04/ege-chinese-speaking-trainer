# Исправление замечаний предсливочного ревью

## Цель

Устранить четыре замечания предсливочного ревью ветки `codex/task-7-migration-strategy`:

- изображения назначенной работы не должны зависеть от последующего редактирования исходного материала;
- удаление аккаунта автора не должно разрушать назначенную работу;
- ссылки подтверждения email и сброса пароля не должны доверять `Origin` или `Host`;
- repository hygiene check должен обнаруживать распространённые форматы приватных ключей.

Проект находится на стадии прототипа, существующих учеников и назначений нет. Обратное заполнение ранее созданных snapshot-assets не требуется.

## Независимые изображения назначения

При создании назначения сервер продолжает сохранять неизменяемый `material_snapshot_json`, но изображения авторского материала копируются в отдельные snapshot-assets до отправки успешного ответа.

Новая таблица `assignment_material_assets` содержит:

- собственный идентификатор;
- ссылку на назначение с `ON DELETE CASCADE`;
- уникальный storage key;
- MIME-тип и размер;
- время создания.

Каждый URL `/api/material-assets/<id>` в снимке заменяется на `/api/assignment-assets/<id>`. Официальные изображения из `/assets/` не копируются, поскольку они версионируются вместе с приложением и не принадлежат авторскому аккаунту.

Копирование выполняется для каждого уникального исходного asset один раз на назначение. Повторяющиеся ссылки внутри материала получают один snapshot URL. Объекты хранятся под ключами `assignments/<assignment-id>/<random>.webp` в том же local/S3 adapter, что и остальные закрытые файлы.

Если копирование или запись метаданных завершается ошибкой, назначение не считается созданным, уже скопированные объекты удаляются, а API возвращает внутреннюю ошибку по существующему общему контракту. Удаление назначения каскадно удаляет метаданные; физические объекты удаляются прикладным кодом в сценариях, где удаление назначения поддерживается.

## Доступ к snapshot-assets

Новый GET endpoint `/api/assignment-assets/{asset_id}` доступен только аутентифицированному пользователю, который:

- является преподавателем назначения; или
- состоит учеником в группе, которой выдано назначение.

Ответ использует сохранённый MIME-тип, `Cache-Control: private, max-age=3600` и `X-Content-Type-Options: nosniff`. Отсутствующий объект или отсутствие прав маскируются одинаковым ответом 404.

Исходный `/api/material-assets/{asset_id}` сохраняет текущую модель доступа. После создания назначения его дальнейшее редактирование, архивирование или удаление вместе с аккаунтом автора не влияет на snapshot-копию.

## Миграции

Добавляется новая Alembic-ревизия после текущего head. Она создаёт `assignment_material_assets` и индекс по `assignment_id` для SQLite и PostgreSQL с одинаковым набором таблиц, колонок и индексов.

Legacy SQLite baseline версий 1–7 остаётся замороженным. Новая схема применяется только через Alembic, как требует стратегия task7.

## Безопасные ссылки аккаунта

`TRAINER_PUBLIC_URL` остаётся единственным источником публичного production URL. Если переменная не задана, ссылка использует фиксированный development fallback `http://127.0.0.1:8080`, не читая `Origin`, `Host` или forwarded-заголовки.

Это сохраняет локальную доставку через outbox и исключает host-header poisoning. Документация явно указывает, что production deployment обязан задавать `TRAINER_PUBLIC_URL`.

## Проверка приватных ключей

Hygiene check заменяет закрытый список из двух заголовков на проверку PEM/OpenSSH marker вида `-----BEGIN ... PRIVATE KEY-----`. Проверка должна обнаруживать как минимум:

- PKCS#8 `PRIVATE KEY` и `ENCRYPTED PRIVATE KEY`;
- `RSA PRIVATE KEY`;
- `EC PRIVATE KEY`;
- `DSA PRIVATE KEY`;
- `OPENSSH PRIVATE KEY`.

Файлы с запрещёнными именами и расширением `.key` продолжают блокироваться независимо от содержимого.

## Тестирование

Разработка выполняется через отдельные red-green циклы:

1. Интеграционный тест создаёт авторский материал, назначает его, переводит исходник в draft и подтверждает доступ ученика к snapshot-изображению.
2. Интеграционный тест удаляет аккаунт автора и подтверждает, что snapshot-изображение всё ещё доступно ученику.
3. Тест доступа подтверждает 404 для постороннего пользователя и успешный ответ для ученика и преподавателя назначения.
4. Миграционные тесты подтверждают совпадение SQLite/PostgreSQL schemas и идемпотентный upgrade.
5. Unit test подтверждает фиксированный fallback URL и отсутствие влияния `Origin`/`Host`.
6. Параметризованный hygiene test проверяет все перечисленные форматы приватных ключей.

После локальных циклов выполняются `make check`, `make test-e2e`, SQLite restore smoke, worker smoke и legacy smoke. Docker/PostgreSQL/S3 проверки выполняются при наличии Docker; без Docker отсутствие этих результатов явно фиксируется перед push.

## Интеграция

Ветка task7 уже включает task6 и почти полностью заменяет task1. После успешной проверки task7 сливается в `main`; task1 и task6 отдельно не сливаются. Единственная дополнительная проверка task1 для отсутствия старых каталогов `data/` и `assets/` переносится в актуальный package-layout test вручную.
