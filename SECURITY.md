# Политика безопасности

Проект обрабатывает аккаунты, учебные назначения и аудиозаписи учеников. Любые такие данные следует считать конфиденциальными.

## Сообщение об уязвимости

Не публикуйте уязвимость, секрет или персональные данные в открытом issue. Сообщите владельцу проекта через доступный вам приватный канал. Публичный адрес для security-контакта в репозитории не задан и не должен быть выдуман.

## Секреты и персональные данные

- Не добавляйте реальные API keys, пароли, SMTP/S3 credentials, session cookies и содержимое `.env` в Git, issue, pull request, лог или тестовую fixture.
- Используйте домены `example.test` и синтетические данные в тестах.
- Не записывайте пароли, токены, cookie, полные заголовки авторизации и содержимое аудио в логи.
- При демонстрации ошибки удаляйте email, IP, идентификаторы аккаунтов и ссылки на закрытые объекты, если они не нужны для диагностики.
- `.env.example` должен содержать только пустые или явно демонстрационные значения.

## Аудиозаписи

- Аудио учеников хранится только в `var/` или закрытом S3/R2 bucket.
- Bucket и объекты не должны быть публичными; доступ выдаётся только приложению с минимально необходимыми правами на нужный bucket/prefix.
- Не копируйте реальные записи в fixtures и не прикладывайте их к issue.
- Временные файлы удаляются после обработки; backup должен иметь не менее строгие ограничения доступа, чем основное хранилище.
- Срок хранения и удаление данных должны соответствовать заявленному пользовательскому сценарию и требованиям владельца сервиса.

## Минимальные права интеграций

- S3/R2 credential: операции только с нужным закрытым bucket и, по возможности, prefix.
- SMTP credential: только отправка с разрешённого адреса, без административного доступа к почтовому аккаунту.
- OpenAI API key: отдельный project key с лимитами и доступом только к необходимой транскрибации.
- PostgreSQL: отдельный пользователь приложения без прав суперпользователя.
- Production-секреты передаются через environment/secret manager и регулярно ротируются.

## Реакция на утечку

1. Прекратите дальнейшую публикацию и сохраните минимальную диагностическую информацию без распространения секрета.
2. Отзовите или ротируйте затронутые ключи, пароли, session secrets и ссылки доступа.
3. Ограничьте доступ к затронутому хранилищу и проверьте audit logs.
4. Удалите секрет из текущего состояния репозитория; необходимость очистки истории согласуйте с владельцем.
5. Проверьте, какие данные и пользователи затронуты, и выполните применимые обязанности по уведомлению.
6. Добавьте тест или контроль, предотвращающий повторение инцидента.

## Security-sensitive области

Расширенная проверка обязательна для:

- `src/trainer/domain/accounts.py`, `src/trainer/infrastructure/database/accounts.py`, auth-контроллеров и session/cookie logic;
- `src/trainer/infrastructure/storage.py`, `audio.py` и маршрутов выдачи записей;
- `src/trainer/infrastructure/mailer.py` и password reset/email verification;
- `src/trainer/infrastructure/database/` и миграций;
- `src/trainer/services/transcription.py`, `src/trainer/infrastructure/transcription.py` и worker;
- Caddy/nginx, Docker Compose, `.env.example` и CI secrets.

Для этих изменений проверяйте авторизацию, разграничение ролей, утечки в логах, path traversal, размер/тип загрузок, CORS/origin, права bucket и поведение при отказе внешней системы.
