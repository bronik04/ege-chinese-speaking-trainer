# Участие в разработке

Архитектурные границы описаны в [docs/architecture.md](docs/architecture.md), правила для автоматизированных агентов — в [AGENTS.md](AGENTS.md), эксплуатация — в [DEVELOPMENT.md](DEVELOPMENT.md).

## Подготовка окружения

```bash
make install
cp .env.example .env
.venv/bin/pre-commit install
```

Замените демонстрационные значения в `.env`. Реальные секреты не добавляйте в Git, issue, логи или тестовые данные.

## Рабочий процесс

1. Создайте отдельную ветку или worktree от актуальной базовой ветки.
2. Внесите одно логически связанное изменение без попутного рефакторинга несвязанных модулей.
3. Добавьте или обновите тесты на изменённое поведение.
4. Запустите Ruff, ESLint, JavaScript- и Python-тесты.
5. При изменении UI запустите Playwright.
6. При изменении схемы БД добавьте миграцию и проверьте создание чистой базы и обновление существующей.
7. При изменении JSON-контента запустите JSON-проверку и профильные тесты.
8. Обновите документацию, если изменились архитектура, конфигурация, deployment или runbook.
9. Просмотрите `git diff` и убедитесь, что в нём нет секретов, runtime-данных и случайных generated-файлов.

## Проверки

```bash
make lint
make test
make check
```

Для пользовательских сценариев:

```bash
npx playwright install chromium
make test-e2e
```

Для диагностики отдельные низкоуровневые команды остаются доступны через `npm run lint:js`, `npm test`, `.venv/bin/python -m ruff check .` и `.venv/bin/python -m coverage report`. Основной локальный quality gate — `make check`.

Docker-конфигурация и image проверяются отдельно:

```bash
make docker-check
make docker-build
```

Минимальное Python-покрытие — 70%. Прошедший процент покрытия не заменяет проверку конкретного нового поведения.

## Изменения базы данных

- PostgreSQL-миграции добавляйте в `migrations/versions/` через Alembic.
- SQLite использует `src/trainer/infrastructure/database/sqlite_migrations.py`; синхронно поддерживайте SQLite и PostgreSQL схемы и добавляйте тест совместимости.
- Не изменяйте существующую применённую миграцию: создавайте следующую версию.
- Проверяйте чистую базу, базу с предыдущей версией и повторный запуск инициализации.

## Ветки и коммиты

Рекомендуемые имена веток: `feature/<topic>`, `fix/<topic>`, `docs/<topic>` или `codex/<topic>`. Рекомендуемый формат сообщений: короткий глагол в повелительной форме, например `Добавить проверку структуры материала`.

Эти соглашения помогают навигации, но не проверяются автоматически. Не создавайте commit и не отправляйте изменения во внешний репозиторий без явного решения владельца проекта.
